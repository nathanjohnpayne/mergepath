## Deploy Tooling

All Firebase projects in this GitHub account use a standardized keyless deployment system. Human maintainers normally authenticate through a shared 1Password-backed GCP ADC source credential, then deploy through short-lived service account impersonation. CI should prefer Workload Identity Federation or another `external_account` credential as the source credential.

### Scripts

| Script | Canonical source | Installed to | Purpose |
|---|---|---|---|
| `gcloud` | `scripts/gcloud/gcloud` in this repo | `~/.local/bin/` | Local wrapper that refreshes access tokens from a 1Password-backed or explicit source credential for normal `gcloud` commands |
| `op-firebase-deploy` | `scripts/firebase/op-firebase-deploy` in this repo | `~/.local/bin/` | Deploy any Firebase project by impersonating its deployer service account |
| `op-firebase-setup` | `scripts/firebase/op-firebase-setup` in this repo | `~/.local/bin/` | One-time setup: creates deployer service account, grants IAM roles, configures impersonation |

Install on a new machine:
```bash
cp scripts/gcloud/gcloud ~/.local/bin/gcloud
cp scripts/firebase/op-firebase-deploy ~/.local/bin/
cp scripts/firebase/op-firebase-setup ~/.local/bin/
chmod +x ~/.local/bin/gcloud ~/.local/bin/op-firebase-deploy ~/.local/bin/op-firebase-setup
```

### Credential Model

- Each project has a dedicated GCP service account: `firebase-deployer@{project-id}.iam.gserviceaccount.com`
- Humans normally read the shared source credential from `op://Private/GCP ADC/credential`
- The 1Password-first deploy-auth model is a deliberate default for repos created from this template and should not be reversed without explicit human approval
- The local `gcloud` wrapper turns that 1Password-backed or explicit source credential into short-lived access tokens so normal `gcloud` commands work without routine browser login prompts
- `op-firebase-deploy` converts that same source credential, or a CI-provided `external_account` credential, into a temporary `impersonated_service_account` credential file at deploy time
- No deploy service-account key is committed to a repo
- Runtime application secrets can still use `op://` references and `op inject` when a repo needs 1Password-managed secrets unrelated to deploy auth

### Usage

```bash
# Set up per-project impersonation
op-firebase-setup {project-id}

# Deploy (auto-detects project from .firebaserc)
op-firebase-deploy

# Deploy explicit project
op-firebase-deploy {project-id}

# Deploy specific targets
op-firebase-deploy --only hosting
op-firebase-deploy {project-id} --only functions

# Optional: activate the project-specific gcloud config
gcloud config configurations activate {project-id}
```

If the shared `Private/GCP ADC` item does not exist yet, bootstrap it once with `gcloud auth application-default login`, then copy the resulting `application_default_credentials.json` into 1Password. After that one-time step, the normal flow returns to 1Password-backed auth.
